Traccia 2 - SQL Injection
Realizzare un attacco di SQL injection [1] di tipo inband, basato su input dell’utente e che utilizzi una o più delle seguenti modalità: Tautologia, commento di fine riga; query piggybacked. 
Mediante l’injection di opportuni comandi mostrare che e’ possibile violare le tre proprietà CIA del database. Nello specifico, supponendo che il database e’ accessibile mediante un applicazione web vulnerabile e che l’attaccante non conosca la struttura delle database a priori:
Raccogliere informazioni sulla struttura del DB o sulla sua configurazione
Esfiltrare dati contenuti in una o piu’ tabelle del database 
Modificare il contenuto del database, ad esempio modifica i valori di un campo all’interno di una tabella
Concellare una o piu’ tabelle, o il loro contenuto

Proporre ed implementare misure di difesa a livello di Database e a livello di applicazione web e mostrare la loro efficacia

Suggerimenti:
Supponete che il database sia accessibile mediante una applicazione web, ad esempio una pagina Php che genera query al DB
Supponete che l’applicazione web sia vulnerabile (ad esempio non usi query parametrizzate o controlli sull’input)
Il database server ed il web server  possono essere emulati mediante  docker container (o virtual machines)
Scegliere versioni del software (OS, DB, Php, etc…) che siano vulnerabili ad attacchi SQLi o disabilitare eventuali meccanismi di protezione.
[1] SQL Injection, https://capec.mitre.org/data/definitions/66.html

SQLi Countermeasures
Because SQLi attacks are so prevalent, damaging, and varied both by attack avenue
and type, a single countermeasure is insufficient. Rather an integrated set of techniques is necessary. In this section, we provide a brief overview of the types of countermeasures that are in use or being researched, using the classification in [SHAR13].
These countermeasures can be classified into three types: defensive coding, detection,
and run-time prevention.
Many SQLi attacks succeed because developers have used insecure coding practices, as we discuss in Chapter 11. Thus, defensive coding is an effective way to dramatically reduce the threat from SQLi. Examples of defensive coding include the following:
• Manual defensive coding practices: A common vulnerability exploited by SQLi
attacks is insufficient input validation. The straightforward solution for eliminating these vulnerabilities is to apply suitable defensive coding practices. An
example is input type checking, to check that inputs that are supposed to be
numeric contain no characters other than digits. This type of technique can
avoid attacks based on forcing errors in the database management system.
Another type of coding practice is one that performs pattern matching to try
to distinguish normal input from abnormal input.
• Parameterized query insertion: This approach attempts to prevent SQLi by
allowing the application developer to more accurately specify the structure
of an SQL query, and pass the value parameters to it separately such that any
unsanitary user input is not allowed to modify the query structure.
• SQL DOM: SQL DOM is a set of classes that enables automated data type validation and escaping [MCCL05]. This approach uses encapsulation of database
queries to provide a safe and reliable way to access databases. This changes the
query-building process from an unregulated one that uses string concatenation
to a systematic one that uses a type-checked API. Within the API, developers
are able to systematically apply coding best practices such as input filtering and
rigorous type checking of user input.
A variety of detection methods have been developed, including the following:
• Signature-based: This technique attempts to match specific attack patterns.
Such an approach must be constantly updated and may not work against selfmodifying attacks.
• Anomaly-based: This approach attempts to define normal behavior then
detect behavior patterns outside the normal range. A number of approaches
have been used. In general terms, there is a training phase, in which the
system learns the range of normal behavior, followed by the actual detection phase.
• Code analysis: Code analysis techniques involve the use of a test suite to detect
SQLi vulnerabilities. The test suite is designed to generate a wide range of SQLi
attacks and assess the response of the system.
Finally, a number of run-time prevention techniques have been developed as
SQLi countermeasures. These techniques check queries at runtime to see if they
conform to a model of expected queries. Various automated tools are available for
this purpose [CHAN11, SHAR13].
